# Authorization-Based Menu Layout & Help Scout Re-Sync — Plan and Progress

Status legend: `[ ]` not started · `[~]` in progress · `[x]` done · `[!]` blocked

---

## 1. Context

The Docusaurus help center was bootstrapped from a Help Scout migration. Articles have since been updated and need to be re-synced. Before touching content, we are implementing an **authorization-based menu layout** so that the sidebar shows only what the viewer is allowed to see:

1. **Role-based** — User, Editor, Admin see different sections.
2. **Org-privilege-based** — A category for a module (Quiz, Survey, KPI, …) is shown only if the viewer's organization has that module enabled.

The main SmartWinnr app at `node_projects/smartwinnr_prd` already encodes the canonical mapping in `modules/common/menu-constants.js` plus org `privileges` in `modules/organizations/server/models/organizations.server.model.js`. The help center should mirror that model, not invent a new one.

## 3. Proposed menu layout

Personas in scope for v1: **User**, **Editor**, **Admin**. Manager / orgadmin / lamadmin / superadmin map to Admin for v1 and can be split out later.

### Top-level sidebar

| Section | User | Editor | Admin | Org-privilege gate | Source dir |
|---|:-:|:-:|:-:|---|---|
| Welcome / Overview | yes | yes | yes | — | `docs/overview` |
| Getting Started | yes | yes | yes | — | `docs/getting-started` |
| Announcements & Release Notes | yes | yes | yes | — | `docs/announcements` |
| Onboarding Guide *(new)* | yes | yes | yes | — | `docs/onboarding` |
| User Guide (My Learning) | yes | yes | yes | — | `docs/user-guide` |
| Manager Guide | hide | yes | yes | `managerView` | `docs/manager-guide` |
| Administrative Guide | hide | yes | yes | (per-module below) | `docs/administration/*` |
| Authoring Tools *(new)* | hide | yes | yes | `authoringTools` | `docs/administration/authoring-tools` |
| Reports & Analytics *(new)* | hide | yes | yes | `learnerReport` OR `adminReports` | `docs/administration/reports` |
| Integrations & API *(new)* | hide | hide | yes | — | `docs/administration/integrations` |
| System Management | hide | hide | yes | — | `docs/administration/system-management` |
| Notifications & Engagement *(new)* | yes | yes | yes | — | `docs/notifications-engagement` |
| Help & Support / FAQs | yes | yes | yes | — | `docs/help-support` |
| Troubleshooting | yes | yes | yes | — | `docs/troubleshooting` |

### UX rationale

- **Role decides sections; org privilege decides modules within a section.** Two-axis gating matches the main app — no surprise menus.
- **User keeps a flat, short menu** (Overview, Getting Started, Onboarding, User Guide, Notifications, Help, Troubleshooting) — no scrolling past admin content.
- **Editor vs Admin diverge only at the top.** Editors get authoring + reports; Admin adds System Management + Integrations. Inside Admin Guide, both see the same module list, filtered by org privileges.
- **Shared rows** (Announcements, Onboarding, Notifications, Help, Troubleshooting) sit outside role gating because they're product-wide.
- **Manager Guide** is shown to Editor/Admin and to anyone whose org has `managerView`, so a future Manager-tier user finds it.

## 4. Implementation phases

### Phase A — Backend: enrich JWT, relax gate, `/api/me`

- [x] **A1.** *(In `smartwinnr_prd`)* `modules/helpAuth/server/controllers/helpAuth.server.controller.js` + `.../models/helpAuth.server.model.js`:
  - Added `tenantId` (ObjectId ref `Organization`) and `privileges` (`[String]`) to `HelpAuthTokenSchema` so the org context survives the send → click → verify gap.
  - Added `loadOrgPrivileges(tenantId)` helper that reads `Organization.privileges` via Mongoose, returns `[]` on miss/error.
  - `sendMagicLink` now stores `tenantId` + `privileges` on the token doc.
  - `verifyMagicLink` adds `orgId` (stringified) and `privileges` to the signed JWT.
  - `generateHandoffToken` is now `async`, loads privileges via `req.user.tenant_id`, signs the same enriched payload.
  - Renamed `PRIVILEGED_ROLES`/`hasPrivilegedRole` → `HELP_CENTER_ROLES`/`hasHelpCenterAccess` and widened to all SmartWinnr roles (user/manager/editor/admin/orgadmin/lamadmin/superadmin), so any signed-in user can request a magic link and the help center decides what they see.
- [x] **A2.** Relaxed `auth/routes.js` — now allows `user`, `manager`, `editor`, `admin`, `orgadmin`, `lamadmin`, `superadmin` (`ALLOWED_ROLES`). Logs and redirects only when none match.
- [x] **A3.** Added `GET /api/me` in `server.js` (after `initAuth(app)`) — returns `{ email, roles, region, orgId, privileges }` from `req.user`. 401s when not authenticated.
- [x] **A4.** Updated `auth/jwt.js` — `signSessionToken` now accepts and signs `orgId` + `privileges`. `auth/middleware.js` populates them on `req.user` for downstream use. `auth/routes.js` callback forwards both fields when re-signing the session cookie.
- [x] **A5.** Updated `/auth/status` to also include `region`, `orgId`, and `privileges`.

### Phase B — Frontend: hydrate user, swizzle sidebar

- [x] **B1.** Created `src/contexts/UserContext.tsx` + `src/theme/Root.tsx`. Root wraps app with `UserProvider`, which fetches `/api/me` on mount, exposes `useCurrentUser()`. Also added `src/access-policy.ts` (single source of truth for `isAllowed(gate, user)`).
- [x] **B2.** Created `src/theme/DocSidebarItem/Category/index.tsx` as a wrap-swizzle. Reads `item.customProps` as `AccessGate`; returns `null` when `isAllowed` fails.
- [x] **B3.** Created `src/theme/DocSidebarItem/Link/index.tsx` as a wrap-swizzle with the same gate for individual article entries.
- [x] **B4.** Annotated `sidebars.ts` with `customProps: {roles, privilege}` on every top-level + module category per the menu layout table. Defined `ALL_ROLES` / `STAFF_ROLES` / `ADMIN_ONLY_ROLES` constants for readability.
- [x] **B5.** Loading UX — `UserContext` defaults to `UNAUTH_USER` (`roles:['user']`, no privileges) until `/api/me` resolves, so the smallest menu shows first and only ever expands. On HTTP 401, redirects to `/auth/login?redirect=<current path>`.
- [x] **B6.** Updated `ChatBot.tsx` to fetch `/api/me` on mount and pick a primary role (`pickPrimaryRole`) to send as `userContext.role`. ChatBot mounts in its own React root via `chatbot-client.js`, outside the Docusaurus tree, so it can't reach `UserContext` — it does its own one-shot fetch.
- **B-note.** Org-privilege gating is currently **disabled** via `PRIVILEGE_GATING_ENABLED = false` in `src/access-policy.ts`. Role gating remains active. All `customProps.privilege` / `customProps.anyPrivilege` annotations on `sidebars.ts` + `_category_.json` files are preserved; re-enable by flipping the flag to `true` (no data migration required).
