# SmartWinnr Help Center: End-to-End Knowledge Hub Architecture

## 1. Content Lifecycle

### 1.1 Article Authoring

Articles enter the Knowledge Hub through three channels:

| Source | Method | Output |
|--------|--------|--------|
| **Manual authoring** | Engineers/writers create `.md` files directly in `docs/{category}/` | Markdown with YAML frontmatter |
| **Freshdesk pipeline** | `scripts/freshdesk/process-freshdesk.js` ingests support ticket CSV, clusters topics, identifies gaps, generates articles via GPT-4o | New `.md` files in `docs/{category}/` |
| **Legacy migration** | `scripts/migrate-helpscout.js` converted HelpScout articles | Already completed; articles live in `docs/` |

### 1.2 Article Format & Storage

All articles are Markdown files stored in `docs/` under 26 category directories (e.g., `administration/`, `getting-started/`, `quiz-assessments/`, `reports/`). Each file has YAML frontmatter:

```yaml
---
title: "Create a New Quiz"               # Action-oriented, Title Case, 5-8 words
description: "Step-by-step guide to..."   # SEO meta description
slug: create-a-new-quiz                   # URL path segment
sidebar_position: 42                      # Navigation ordering
last_update:
  date: 2026-03-06
  author: "Freshdesk Analysis"            # or manual author name
---
```

**Style rules** (from `SmartWinnr-Help-Style-Guide.md`):
- American English, active voice, present tense
- Bold all UI elements (`**Save**`, `**Submit**`)
- Numbered steps for procedures, bullets for options
- 15-20 word max sentences
- `> **Tip**:` and `> **Note**:` callout boxes

**Directory structure** (26 categories, 877+ files):
```
docs/
  overview/            # Platform overview
  getting-started/     # Onboarding, account setup
  administration/      # Admin tasks (largest: user mgmt, config, setup)
  quiz-assessments/    # Quiz module
  learning-smartpaths/ # Learning paths
  coaching-performance/# Coaching features
  competitions-gamification/
  reports/             # Reporting
  surveys-feedback/
  forms-data-collection/
  knowledge-hub/
  microlearning/
  mobile-platform-tools/
  troubleshooting/
  ... (26 total)
```

### 1.3 Navigation & Sidebar

`sidebars.ts` uses Docusaurus `autogenerated` type -- directories automatically become sidebar categories. No manual sidebar registration needed when adding new articles. Categories:
- **Overview** and **Getting Started** (expanded by default)
- **Administrative Guide** (collapsed, 11 subcategories: system-management, quiz-module, smartpath, video-coaching, etc.)
- **User Guide** (collapsed, includes user-guide, manager-guide, help-support, troubleshooting)

### 1.4 Build & Deployment

```
Author writes .md → Git commit → GitLab CI (lint + build) → Railway deployment
```

- `npm run build` compiles Docusaurus markdown into static HTML/JS/CSS in `build/`
- `server.js` (Express) serves the static build AND the API endpoints on a single port (3000)
- Railway pulls latest Git commit, builds Docker image (`Dockerfile.docusaurus`), deploys
- Post-deployment: `npm run index-internal` re-indexes docs into ChromaDB

## 2. ChromaDB Vector Storage Architecture

### 2.1 What ChromaDB Stores

ChromaDB holds a single collection called `smartwinnr_docs`. Each document entry contains:

| Field | Content | Purpose |
|-------|---------|---------|
| `id` | `doc_<base64(relativePath)>` | Stable ID across machines |
| `documents` | Raw markdown content (frontmatter stripped) | Full-text for retrieval context |
| `embeddings` | 1536-dim float array from `text-embedding-3-small` | Cosine similarity search |
| `metadatas.title` | Extracted from first `# Heading` or filename | Display in search results |
| `metadatas.source` | Relative file path (e.g., `administration/create-user.md`) | Navigation link generation |
| `metadatas.url` | Derived URL path (e.g., `/administration/create-user`) | Frontend routing |
| `metadatas.lastModified` | File mtime ISO timestamp | Freshness tracking |
| `metadatas.contentHash` | SHA-256 of markdown content | Incremental change detection |
| `metadatas.type` | Always `"documentation"` | Collection filtering |

### 2.2 Ingestion Pipeline (How Articles Enter ChromaDB)

**Script**: `scripts/internal-indexer.js` (`InternalIndexer` class)
**Trigger**: `npm run index-internal` (post-deployment or manual)
**Env**: `FORCE_FULL_REINDEX=true` for complete rebuild

**Step-by-step ingestion flow**:

```
1. SCAN: Recursively find all .md/.mdx files in docs/
   └─ getDocumentFiles('docs/') → 877 file paths

2. PROCESS each file:
   ├─ Read file content
   ├─ Extract title from first # heading (or use filename)
   ├─ Strip YAML frontmatter for clean content
   ├─ Generate SHA-256 content hash
   ├─ Compute relative path → base64-encoded document ID
   └─ Build metadata object {title, source, url, lastModified, contentHash, type}

3. CHANGE DETECTION (incremental indexing):
   ├─ Fetch existing docs from ChromaDB: collection.get({include: ['metadatas']})
   ├─ Compare contentHash of each file against stored hash
   ├─ Classify: NEW (not in ChromaDB), CHANGED (hash differs), UNCHANGED (skip), DELETED (in ChromaDB but gone from disk)
   └─ Only process NEW + CHANGED documents (massive efficiency gain)

4. EMBED (batch size: 10 documents):
   ├─ For each document, call POST /api/vector/embed with document content
   │   └─ server.js forwards to OpenAI: POST https://api.openai.com/v1/embeddings
   │       ├─ Model: text-embedding-3-small (1536 dimensions)
   │       ├─ Input: full markdown content string
   │       └─ Returns: float[1536] embedding vector
   ├─ Retry with exponential backoff (2s, 4s, 6s) on failure
   └─ 30-second timeout per embedding call

5. UPSERT to ChromaDB:
   collection.upsert({
     documents: [content1, content2, ...],     // Raw markdown text
     embeddings: [vec1, vec2, ...],            // Pre-computed 1536-dim vectors
     metadatas: [meta1, meta2, ...],           // Title, source, url, hash
     ids: [id1, id2, ...]                      // base64 path IDs
   })

6. DELETE removed docs:
   collection.delete({ ids: [deletedId1, ...] })
```

**Key design decisions**:
- Embeddings are pre-computed externally (not ChromaDB's built-in embedding function) -- this allows using OpenAI's `text-embedding-3-small` model
- Document IDs use base64-encoded relative paths for cross-machine consistency
- Content hashing enables incremental updates: only new/changed docs get re-embedded (saves OpenAI API costs)
- The indexer calls the same `/api/vector/embed` endpoint that the server exposes, running on localhost

### 2.3 ChromaDB Connection

```javascript
// server.js and internal-indexer.js both use:
const chromaClient = new ChromaClient({
  host: process.env.CHROMA_HOST || 'localhost',  // Railway service name or external host
  port: Number(process.env.CHROMA_PORT || 8000),
  ssl: process.env.CHROMA_SSL === 'true'
});
const COLLECTION_NAME = process.env.COLLECTION_NAME || 'smartwinnr_docs';
```

### 2.4 How ChromaDB Enables Semantic Search

ChromaDB stores each document as a point in 1536-dimensional vector space. When a user query arrives:

1. The query text is converted to a 1536-dim vector using the same `text-embedding-3-small` model
2. ChromaDB performs **cosine similarity search** across all stored document vectors
3. Returns the `nResults` nearest neighbors, ranked by distance (lower = more similar)
4. Distance threshold: `< 0.8` indicates a high-confidence match (used for citation eligibility)

This means queries like *"how do I set up a leaderboard"* will match documents about competitions and gamification even if they don't contain the exact word "leaderboard" -- the embedding captures semantic meaning.

## 3. Chatbot RAG (Retrieval-Augmented Generation) Workflow

### 3.1 Architecture Overview

```
User Question
     │
     ▼
┌─────────────────────────────────────────────┐
│  POST /api/chat  (server.js)                │
│                                              │
│  1. RETRIEVE: searchDocuments(question)       │
│     ├─ Embed question → text-embedding-3-small│
│     ├─ ChromaDB cosine search (top 5 docs)   │
│     └─ Return docs + distances + metadata    │
│                                              │
│  2. AUGMENT: Build context prompt            │
│     ├─ Concatenate top 5 doc contents (500ch)│
│     ├─ Extract citations (distance < 0.8)    │
│     └─ Inject into system prompt             │
│                                              │
│  3. GENERATE: OpenAI Chat Completion         │
│     ├─ Model: gpt-3.5-turbo                  │
│     ├─ System: context + guidelines          │
│     ├─ User: original question               │
│     ├─ Temperature: 0.7                      │
│     └─ Max tokens: 500                       │
│                                              │
│  4. RESPOND: Format + store                  │
│     ├─ AI message + top 3 citations          │
│     ├─ Confidence score (0.8 if citations)   │
│     └─ Store in conversations Map            │
└─────────────────────────────────────────────┘
     │
     ▼
User receives answer + source article links
```

### 3.2 Detailed Step-by-Step Flow

**Step 1 -- Question Intake** (`POST /api/chat`):
```json
{
  "message": "How do I create a quiz with passing score?",
  "conversationId": "uuid-or-null",
  "userContext": {}
}
```
If `conversationId` is null, a new UUID is generated. Conversation history is loaded from an in-memory `Map`.

**Step 2 -- Document Retrieval** (`searchDocuments(query, limit=5)`):
```javascript
// 2a. Embed the user's question
const embeddingResponse = await axios.post('http://localhost:3000/api/vector/embed', {
  text: "How do I create a quiz with passing score?",
  model: 'text-embedding-3-small'
});
// Returns: { embedding: [0.023, -0.156, 0.089, ...] }  // 1536 floats

// 2b. Search ChromaDB
const collection = await chromaClient.getCollection({ name: 'smartwinnr_docs' });
const results = await collection.query({
  queryEmbeddings: [queryEmbedding],
  nResults: 5,
  include: ['documents', 'metadatas', 'distances']
});
// Returns: top 5 documents ranked by cosine distance
```

**Step 3 -- Context Building**:
```javascript
// Build context string from search results
context = results.map((doc, i) => `Document ${i+1}:
Title: ${doc.metadata.title}
Content: ${doc.content.substring(0, 500)}...
---`).join('\n\n');

// Extract citations (only high-confidence matches)
citations = results
  .filter(doc => doc.distance < 0.8)
  .map(doc => ({
    title: doc.metadata.title,
    url: doc.metadata.url,
    snippet: doc.content.substring(0, 150) + '...',
    source: doc.metadata.source
  }));
```

**Step 4 -- AI Response Generation**:
```javascript
const systemPrompt = `You are a helpful SmartWinnr Help Assistant...
Use the following context from the SmartWinnr documentation to answer:
${context}
Guidelines:
- Be helpful, concise, and accurate
- Reference specific sections or features when relevant
- If context doesn't contain relevant information, say so
- Provide actionable advice when possible
- Use a friendly, professional tone`;

const response = await axios.post('https://api.openai.com/v1/chat/completions', {
  model: 'gpt-3.5-turbo',
  messages: [
    { role: 'system', content: systemPrompt },
    { role: 'user', content: userQuestion }
  ],
  temperature: 0.7,
  max_tokens: 500
});
```

**Step 5 -- Response Formatting**:
```json
{
  "conversationId": "uuid-1234",
  "response": {
    "message": "To create a quiz with a passing score, navigate to...",
    "citations": [
      { "title": "Create a New Quiz", "url": "/quiz-assessments/create-quiz", "snippet": "...", "source": "..." }
    ],
    "relatedLinks": [...],
    "confidence": 0.8
  }
}
```

**Step 6 -- Conversation Storage**:
Both user and assistant messages are stored in an in-memory `Map<conversationId, Message[]>`. This enables multi-turn conversations within a session but is lost on server restart.

### 3.3 Vector Search (Non-Chat)

The same retrieval pipeline is also exposed for direct search via `POST /api/vector/search`:

```json

// Response
{
```

## 4. Freshdesk Content Pipeline (Implemented)

The Freshdesk pipeline feeds new articles into the Knowledge Hub. It is already fully implemented in `scripts/freshdesk/`.

### Pipeline Flow

```
Freshdesk CSV (605 tickets)
  │
  ▼
Stage 1: ingest-csv.js ─── Clean email threads, strip signatures/GDPR/CIDs, dedup
  │                         Output: cleaned-queries.json (381 unique queries)
  ▼
Stage 2: cluster-queries.js ── Rule-based clustering by Type field + keyword matching
  │                             AI fallback via Claude for uncategorized queries
  │                             Output: clusters.json (324 clusters)
  ▼
Stage 3: gap-analysis.js ──── Compare clusters against 877 existing docs
  │                            Fuzzy title match + keyword overlap scoring
  │                            Output: gap-report.json (218 GAP, 102 PARTIAL, 4 COVERED)
  ▼
Stage 4: generate-articles.js ── GPT-4o generates markdown per style guide
  │                                New articles + enhancement patches
  ▼
Stage 5: validate-articles.js ── Check frontmatter, titles, sentence length, UI term bolding
  │
  ▼
Stage 6: prepare-output.js ──── Copy to docs/, generate recommendations + summary
```

### Post-Pipeline: Making New Articles Searchable

After the Freshdesk pipeline places new `.md` files in `docs/{category}/`:

1. **Git commit** the new articles
2. **Deploy** to Railway (triggers Docker rebuild + `npm run build`)
3. **Run indexer**: `npm run index-internal`
   - Scans `docs/`, finds new files (no matching contentHash in ChromaDB)
   - Embeds each new article via `text-embedding-3-small`
   - Upserts to ChromaDB `smartwinnr_docs` collection
4. **Immediately searchable**: chatbot and vector search now return new articles

There is **no automated re-indexing trigger** -- indexing must be run manually after deployment. A post-deploy script or Railway deploy hook could automate this.

## 5. Data Update & Refresh Cycles

| Event | What Happens | Indexing Impact |
|-------|-------------|-----------------|
| New article added to `docs/` | File detected as NEW by indexer | Embedded + upserted on next `npm run index-internal` |
| Article content edited | Hash changes → detected as CHANGED | Re-embedded + upserted |
| Article deleted | Missing from disk → detected as DELETED | Removed from ChromaDB |
| No changes | Hash matches → UNCHANGED | Skipped (no API cost) |
| Full reset needed | `FORCE_FULL_REINDEX=true npm run index-internal` | Deletes collection, re-embeds all 877+ docs |

**Cost efficiency**: Incremental indexing only calls OpenAI embeddings API for new/changed docs. At ~$0.02/1M tokens, indexing all 877 docs costs roughly $0.10-0.50 depending on content length.

## 10. Ensuring Accurate, Up-to-Date Answers

The combination of Knowledge Hub + ChromaDB + Chatbot ensures accuracy through:

1. **Single source of truth**: All articles live as `.md` files in `docs/` under Git version control. ChromaDB is a derived index, not a primary store.

2. **Content-hash change detection**: The indexer uses SHA-256 hashes to detect edits. Changed articles are automatically re-embedded on next index run -- stale vectors don't persist.

3. **Semantic retrieval over keyword matching**: ChromaDB's cosine similarity search finds relevant docs even when users phrase questions differently from article titles. A question about "scoring in competitions" matches articles titled "Configure Leaderboards" because the embeddings capture semantic proximity.

4. **Grounded generation**: The chatbot's GPT-3.5-turbo response is always grounded in retrieved documentation context. The system prompt instructs: *"If the context doesn't contain relevant information, say so."* This prevents hallucination.

5. **Citation transparency**: Every chatbot response includes up to 3 citation links to source articles (distance < 0.8 threshold). Users can verify answers against the original documentation.

6. **Freshdesk feedback loop**: Support tickets reveal documentation gaps. The Freshdesk pipeline systematically identifies uncovered topics (218 GAPs from 605 tickets) and generates articles to fill them. This closes the loop: user confusion → support ticket → new article → indexed → chatbot can answer next time.

7. **Incremental efficiency**: Only new/changed documents incur embedding API costs. Full re-index is available via `FORCE_FULL_REINDEX=true` but rarely needed.

## 11. Local Development Setup

ChromaDB is already installed locally. Here is the complete workflow for running the entire system on a local machine.

### 11.1 Prerequisites

- **Node.js** >= 20.0.0
- **ChromaDB** installed locally (runs on `localhost:8000`)
- **OpenAI API key** set in `.env` file

### 11.2 Environment Configuration

The `.env` file (already present) configures local development:

```bash
# .env (local development defaults)
NODE_ENV=development
OPENAI_API_KEY=sk-proj-...          # Your OpenAI key
CHROMA_HOST=localhost                # ChromaDB on same machine
CHROMA_PORT=8000                     # Default ChromaDB port
COLLECTION_NAME=smartwinnr_docs      # Vector collection name
EMBEDDING_MODEL=text-embedding-3-small
CHAT_MODEL=gpt-4o-mini              # Or gpt-3.5-turbo for lower cost
```

### 11.3 Starting the Local Stack

**Terminal 1 -- Start ChromaDB:**
```bash
chroma run --host localhost --port 8000
```
ChromaDB persists data locally. The `smartwinnr_docs` collection is created automatically on first index run.

**Terminal 2 -- Start the unified server (docs + API):**
```bash
# Option A: Build docs first, then serve production-like
npm run build && node server.js
# Serves at http://localhost:3000 (docs + API on same port)

# Option B: Development mode with hot reload (docs only, no API)
npm run dev
# Serves docs at http://localhost:3001 (no chatbot API)

# Option C: Full dev mode (docs + chatbot API in parallel)
npm run dev:full
# Docs at http://localhost:3000, ChatBot API at http://localhost:3002
```

**For local development with chatbot working**, use Option A or C. The chatbot requires the Express server (`server.js`) to be running because it serves the `/api/chat` and `/api/vector/search` endpoints.

### 11.4 Indexing Documents Locally

After ChromaDB and the server are running:

```bash
# Incremental index (only new/changed docs since last run)
npm run index-internal

# Full re-index (delete collection + re-embed all 877+ docs)
FORCE_FULL_REINDEX=true npm run index-internal
```

**Important**: The indexer calls `http://localhost:{PORT}/api/vector/embed` to generate embeddings. The Express server (`server.js`) must be running first. The PORT defaults to 3000 locally (via `.env` or the server default).

**What happens during indexing:**
1. Indexer scans all `.md`/`.mdx` files in `docs/`
2. For each file: extracts content, strips frontmatter, generates SHA-256 hash
3. Compares hashes against existing ChromaDB entries
4. Only new/changed files are sent to OpenAI for embedding (saves cost)
5. Embeddings are upserted into ChromaDB `smartwinnr_docs` collection
6. Deleted files are removed from ChromaDB

**First-time indexing** will process all 877+ documents. Expect ~5-10 minutes and ~$0.10-0.50 in OpenAI embedding costs.

### 11.5 Running the Freshdesk Pipeline Locally

```bash
# Dry run (no article generation, no API keys needed for this)
node scripts/freshdesk/process-freshdesk.js --dry-run

# Full pipeline with article generation (requires OPENAI_API_KEY)
node scripts/freshdesk/process-freshdesk.js --min-cluster 3

# With AI clustering (requires ANTHROPIC_API_KEY too)
ANTHROPIC_API_KEY=sk-ant-... node scripts/freshdesk/process-freshdesk.js

# Use existing intermediate data (skip re-ingestion)
node scripts/freshdesk/process-freshdesk.js --skip-ingest --skip-cluster
```

The pipeline auto-detects CSV files in `freshdesk/` directory. Output goes to `scripts/freshdesk/output/` and intermediate data to `data/` (both gitignored).

### 11.6 End-to-End Local Workflow

Complete cycle from Freshdesk tickets to searchable chatbot answers:

```bash
# 1. Start ChromaDB
chroma run --host localhost --port 8000

# 2. Start the server (separate terminal)
npm run build && node server.js

# 3. Run Freshdesk pipeline to generate new articles
node scripts/freshdesk/process-freshdesk.js --dry-run     # Review first
node scripts/freshdesk/process-freshdesk.js --min-cluster 3  # Generate

# 4. New articles are now in scripts/freshdesk/output/new-articles/
```

### 11.8 Local vs Production Differences

| Aspect | Local | Production (Railway) |
|--------|-------|---------------------|
| ChromaDB | `localhost:8000` | Railway internal service |
| OpenAI key | `.env` file | Railway env vars / AWS Secrets Manager |
| Port | 3000 (default) | Set by Railway (`PORT` env) |
| Data persistence | ChromaDB local storage | Railway volume |
| Conversation storage | In-memory (lost on restart) | Same (in-memory) |
| SSL | None | Railway provides HTTPS |
| Indexing trigger | Manual `npm run index-internal` | Manual post-deploy |
