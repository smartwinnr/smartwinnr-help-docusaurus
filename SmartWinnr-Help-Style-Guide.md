# SmartWinnr Help Document Style Guide

*American English, user-friendly, professional*

## 1. Tone & Voice

- **Friendly but professional**: Approachable and supportive, without being overly casual.
- **Encouraging & solution-oriented**: Focus on what the user can do.
- **Clear and direct**: Avoid unnecessary jargon; explain technical terms in plain language.

## 2. Header Guidelines

Every help document should have one main title (H1) and optional subheadings (H2, H3) that guide the reader.

### Main Title (H1) Rules

- **Be action-oriented**: Start with a verb that reflects the user's goal.
  - ✅ "Upload a Training Video"
  - ❌ "Video Uploading Process"

- **Avoid internal jargon**: Use the words users would type into search.
  - ✅ "Reset Your Password" instead of "Authentication Credential Reset"

- **Be specific**: Clearly state what the guide is about, avoiding vague terms.
  - ✅ "View Learner Scores in Coaching Analytics"
  - ❌ "Coaching Analytics Overview"

- **Keep it short**: Aim for 5–8 words if possible.

- **Capitalize Each Major Word (Title Case)**.

### Subheading (H2 / H3) Rules

- Describe sections in a question or task format:
  - "How to Add a New User"
  - "Steps to Download Your Report"

- Keep them scan-friendly — users should know what's inside the section at a glance.

- Use parallel structure:
  - If one heading starts with "How to…," keep the others similar ("How to…," "How to…").

## 3. Structure

- **Headings & Subheadings**: Clear, descriptive, following the rules above.
- **Step-by-step instructions**: Numbered lists for sequences; bullet points for options.
- **Highlight key actions**: Bold important buttons, menu names, or commands.
  - Example: Click **Save** to confirm.
- **Screenshots or visuals**: Include with short, clear captions.

## 4. Language Rules

- **American English spelling** (e.g., "customize," not "customise").
- **Present tense, active voice**:
  - ✅ "Click the button to start"
  - ❌ "The button should be clicked to start"
- **Short sentences**: 15–20 words max.
- **Positive framing**: "Remember to…" instead of "Don't forget to…"
- **Consistent terminology**: Same word for the same feature across all docs.

## 5. Formatting Conventions

- **UI elements**: Use bold for labels, buttons, menus, and field names.
- **Code or commands**: Use inline code formatting.
- **Links**: Use descriptive text instead of "click here."
  - Example: [View your dashboard](link-url)

## 7. Common Phrasing Conventions

- Use "issue" or "problem" instead of "error" when possible.
- Use "need to" or "should" instead of "must" for a softer tone.
- Replace technical shorthand with friendly equivalents unless the audience is technical.
